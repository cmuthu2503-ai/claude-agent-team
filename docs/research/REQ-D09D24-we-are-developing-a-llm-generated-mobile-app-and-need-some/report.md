## 1. Executive Summary

Your team's core challenge — screens generated by LLMs that lack accessibility considerations, especially font scaling — can be addressed by building a **three-layer Accessibility Test Agent** that combines:

1. **Static code analysis** via LLM-powered linting
2. **Runtime visual testing** via Appium + device font-scaling automation
3. **An LLM-powered correction loop** that auto-generates fix suggestions

Recent research (AccessGuru, ASSETS '25) demonstrates that LLMs with corrective re-prompting achieve an **84% violation score reduction** on accessibility issues — significantly outperforming standalone tools. The recommended approach is a hybrid agent that orchestrates existing accessibility engines (Axe DevTools Mobile, Android Accessibility Scanner) alongside a multimodal LLM for the visual and semantic checks that rule-based tools cannot catch.

## 2. Scope & Background

**Research scope:** Architecture and implementation plan for an LLM-powered Accessibility Test Agent focused on mobile apps, with emphasis on font scaling, Dynamic Type, and layout resilience. Covers agent architecture, tool landscape, WCAG compliance requirements, and CI/CD integration.

**Why this matters:**

- **WCAG 1.4.4 (Resize Text)** requires apps to support 200% text scaling without content loss — a Level AA legal requirement.
- **94.8%** of top websites fail accessibility guidelines (WebAIM 2025); mobile apps fare worse.
- **LLM-generated code typically ignores font scaling** because models default to fixed constraints and hardcoded dimensions — a training-data bias problem.
- **ADA / Section 508 lawsuits** increasingly target mobile experiences.
- **Manual testing** of font scaling across 5+ scale levels × dozens of screens is prohibitively expensive.

> **Sources:** Live web research performed April 8, 2026. Primary sources include Deque Docs, AccessGuru (ASSETS '25), QA Wolf, A11y Pros, Percy/BrowserStack, and various AI testing tool guides.

## 3. Key Findings

| # | Finding | Detail | Confidence |
|---|---------|--------|------------|
| 1 | **LLM-generated mobile code systematically lacks font-scaling support** | LLMs produce fixed-height containers, hardcoded `dp` values instead of `sp`, and omit `numberOfLines=0` / `wrap_content` properties. This is a training-data bias — most code examples don't demonstrate accessibility-first patterns. | High |
| 2 | **Combined LLM + rule-based detection achieves 84% violation reduction** | AccessGuru (ASSETS '25) showed that pairing Axe-Playwright with an LLM semantic detector and corrective re-prompting far outperforms standalone prompting (max 50%). This pattern applies directly to mobile. | High |
| 3 | **Font-scaling testing can be fully automated** | Appium can programmatically change device font scale via `adb shell settings put system font_scale 1.5` (Android) and XCUITest capabilities (iOS). Combined with Percy/Applitools visual comparison, this enables automated screenshot diffing at multiple scale levels. | High |
| 4 | **Axe DevTools Mobile supports CI/CD integration** | Deque's `AxeUiAutomator2` and `AxeXCUITest` drivers execute `mobile: axeScan` commands inline with Appium tests, generating structured accessibility violation reports. | High |
| 5 | **Multimodal LLMs detect visual/layout violations that rule engines miss** | Semantic issues — truncated text, overlapping elements, unreadable text at scale — require visual reasoning. Vision-capable models (GPT-4o, Claude) analyze screenshots and detect layout breakage no rule-based tool catches. AccessGuru detected 104 additional semantic violations that Axe-Playwright missed. | High |
| 6 | **Three violation categories need different detection strategies** | Per AccessGuru taxonomy: **Syntactic** (missing attributes) → rule engines; **Layout** (contrast, scaling) → rule engines + visual; **Semantic** (meaningful content) → LLM required. Font scaling spans layout and semantic categories. | High |
| 7 | **Self-healing agentic test tools reduce maintenance by ~70%** | AI-powered tools like Maestro, Drizz, and testRigor use vision-based element location instead of brittle selectors — critical since LLM-generated screens change frequently. | Medium |

## 4. Analysis

### 4.1 Pros of the Recommended Architecture

| Advantage | Explanation |
|-----------|-------------|
| **Catches what static analysis can't** | Font-scaling issues only manifest at runtime — overlapping text, truncated labels, and content pushed off-screen are visual problems requiring screenshot analysis. |
| **Auto-correction loop** | The LLM can both detect AND generate code fixes. AccessGuru achieves 77% similarity to human developer corrections. |
| **Integrates into existing CI/CD** | Appium + Axe DevTools Mobile already have CI/CD hooks; adding the LLM layer is an orchestration task, not a re-architecture. |
| **Reduces manual QA bottleneck** | Automated runs across 5 font-scale levels × all screens replaces hours of manual testing per sprint. |
| **Shifts left** | Static analysis of generated code catches ~60% of issues before they even reach a device. |

### 4.2 Cons and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| **LLM API costs** | ~$0.01–0.03 per screenshot analysis adds up | Run full analysis only on changed screens; use hash-based change detection. |
| **False positives in visual analysis** | LLMs may flag acceptable layout reflows | Human-in-the-loop review for first pass; build a training corpus over time. |
| **iOS Dynamic Type complexity** | iOS simulator font scaling requires XCUITest launch arguments and doesn't hot-reload | Use real device clouds (BrowserStack / Sauce Labs). |
| **Generated fix quality** | LLM corrections may break other functionality | Run corrections through the full test suite before auto-merging. |
| **Pipeline latency** | Full agent run per PR could take 10–20 minutes | Parallel execution and incremental testing. |

### 4.3 Alternatives Considered

| Alternative | Verdict |
|-------------|---------|
| **Manual QA only** | Not scalable with LLM-generated screens that change frequently. |
| **Rule-based tools only** (Axe, Accessibility Scanner) | Miss visual/layout issues from font scaling entirely; cannot detect truncated text or overlapping elements. |
| **Vision-only AI testing** (Drizz, Applitools) | Detects visual regressions but lacks accessibility-specific WCAG knowledge. |
| **AccessGuru as-is** (web-focused) | Excellent research foundation but web-only; requires adaptation for mobile XML/SwiftUI layouts. |

## 5. Agent Architecture — Three-Layer Design

### 5.1 Layer Overview

| Layer | Purpose | Tools | Trigger | Catches |
|-------|---------|-------|---------|---------|
| **L1 — Static Code Analysis** | Lint generated code for a11y anti-patterns | LLM code review + Android Lint + SwiftLint + custom rules | Pre-commit / PR creation | Fixed heights, `dp` instead of `sp`, missing `numberOfLines=0`, hardcoded constraints, missing `adjustsFontForContentSizeCategory` |
| **L2 — Runtime Rule-Based Scan** | Automated WCAG rule checks at multiple font scales | Appium + Axe DevTools Mobile (`mobile: axeScan`) + ADB font-scale commands | CI pipeline, per screen | Touch target sizes, color contrast, missing labels, ARIA violations, scrollability |
| **L3 — Visual-Semantic AI Analysis** | Screenshot comparison + LLM reasoning for layout violations | Appium screenshots + Percy/Applitools + Multimodal LLM (GPT-4o / Claude) | CI pipeline, per screen × per scale level | Text truncation, overlapping elements, content pushed off-screen, unreadable text, broken layouts |

### 5.2 Component Comparison Matrix

| Criteria | Weight | L1 Only | L2 Only | Full Agent (L1+L2+L3) |
|----------|--------|---------|---------|------------------------|
| Font-scaling detection | High | ⚠️ 3/10 | ⚠️ 5/10 | ✅ 9/10 |
| Auto-fix generation | High | ✅ 8/10 | ❌ 2/10 | ✅ 9/10 |
| CI/CD integration | Medium | ✅ 9/10 | ✅ 7/10 | ⚠️ 6/10 |
| Cost per run | Medium | ✅ 9/10 | ✅ 7/10 | ⚠️ 5/10 |
| False positive rate | Medium | ⚠️ 5/10 | ✅ 8/10 | ⚠️ 6/10 |
| Coverage breadth | High | ⚠️ 4/10 | ⚠️ 6/10 | ✅ 9/10 |
| **Weighted Score** | | **5.5 / 10** | **5.8 / 10** | **7.8 / 10** |

### 5.3 Core Agent Loop — Detect → Correct → Verify

The agent follows an AccessGuru-inspired iterative cycle:

1. **Detect** — Run Axe DevTools Mobile scan + capture screenshots at font scales **1.0×, 1.3×, 1.5×, 2.0×**.
2. **Analyze** — Feed screenshots + view hierarchy XML to a multimodal LLM with a prompt taxonomy covering truncation, overlap, off-screen content, and unreadable text.
3. **Correct** — LLM generates code fixes for the generated source files using role-play persona + contextual prompting + corrective re-prompting (as validated by AccessGuru).
4. **Verify** — Re-run scan on corrected code to confirm the violation score decreased; reject corrections that introduce new issues.

## 6. Recommendation

**Build the Full Three-Layer Agent (L1 + L2 + L3) in a phased rollout:**

| Phase | Content | Duration |
|-------|---------|----------|
| **Phase 1** | L1 — Static code analysis | Weeks 1–2 |
| **Phase 2** | L2 — Appium + Axe scans at multiple font scales | Weeks 2–4 |
| **Phase 3** | L3 — Multimodal LLM visual analysis with correction loop | Weeks 4–7 |
| **Phase 4** | CI/CD gating + false-positive tuning | Weeks 6–8+ |

This delivers immediate value from day one while building toward comprehensive coverage.

### When NOT to Follow This Recommendation

- Your team ships **fewer than 2 releases/month** — manual QA may be sufficient.
- Your LLM-generated screens use a **design system with pre-built accessible components** — L1 alone may catch 80%+ of issues.
- Budget is **severely constrained** (< $500/month for tooling) — start with L1 (free) + Android Accessibility Scanner (free).

## 7. Implementation Roadmap

| Step | Owner | Timeline | Details |
|------|-------|----------|---------|
| **1. Define accessibility rule corpus** | QA Lead + Accessibility SME | Week 1 | Create a taxonomy of font-scaling violations specific to your tech stack (React Native / Swift / Kotlin). Use Deque's text-scaling guide as baseline. |
| **2. Build L1 — Static Code Linter** | Platform Engineer | Weeks 1–2 | LLM prompt reviewing generated code for: fixed heights, `dp` text sizes, missing `wrap_content`, missing `ScrollView` wrappers, missing `adjustsFontForContentSizeCategory`. Run as a PR check. |
| **3. Set up Appium + Axe DevTools Mobile** | QA Automation Engineer | Weeks 2–4 | Configure Appium with `AxeUiAutomator2` / `AxeXCUITest` drivers. Script font-scale changes (`adb shell settings put system font_scale X`). Add `mobile: axeScan` at each scale level. |
| **4. Build screenshot capture pipeline** | QA Automation Engineer | Weeks 3–4 | Capture screenshots at 1.0×, 1.3×, 1.5×, 2.0× for every screen. Store as CI artifacts. Integrate Percy or Applitools for baseline comparison. |
| **5. Build L3 — Multimodal LLM Analyzer** | ML / Platform Engineer | Weeks 4–6 | Send screenshot pairs (baseline vs. scaled) + view hierarchy to GPT-4o / Claude with AccessGuru-style prompts. Parse structured violation output. Generate fix suggestions. |
| **6. Implement correction loop** | Platform Engineer | Weeks 5–7 | Apply LLM-suggested fixes → rebuild → re-scan → compare violation scores. Accept only improvements. Surface results in PR comments. |
| **7. Integrate into CI/CD** | DevOps | Weeks 6–8 | L1 blocks merge; L2 + L3 produce warnings initially, become blocking after tuning. |
| **8. Tune and reduce false positives** | QA Team | Ongoing | Review flagged violations weekly. Build ignore-list for acceptable patterns. Track precision/recall metrics. |
