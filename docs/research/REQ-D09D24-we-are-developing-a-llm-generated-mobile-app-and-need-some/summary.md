# Executive Summary

## Accessibility Test Agent for LLM-Generated Mobile Apps

**Date:** April 8, 2026

Mobile screens generated by LLMs consistently fail accessibility standards — particularly font scaling (WCAG 1.4.4, which mandates 200% text scaling without content loss). The root cause is a training-data bias: LLMs default to fixed-height containers and hardcoded dimensions because most source code examples lack accessibility-first patterns. Manual testing across multiple scale levels and dozens of screens is prohibitively expensive and unsustainable.

**Our recommendation is to build a three-layer Accessibility Test Agent.** Layer 1 performs static code analysis using an LLM-powered linter that catches anti-patterns (e.g., `dp` instead of `sp`, fixed heights, missing `wrap_content`) at PR time. Layer 2 runs automated runtime scans using Appium and Axe DevTools Mobile across font scales of 1.0×, 1.3×, 1.5×, and 2.0×. Layer 3 — the differentiator — feeds screenshots and view hierarchies to a multimodal LLM that detects visual violations rule-based engines cannot: truncated text, overlapping elements, and content pushed off-screen. The agent then generates code fix suggestions and re-verifies that violation scores decrease before surfacing results in pull request comments.

This approach is grounded in peer-reviewed research. The AccessGuru system (ASSETS '25) demonstrated that combining rule-based engines with LLM-powered semantic analysis achieves an **84% accessibility violation score reduction** — nearly double the effectiveness of standalone tools. The research also showed that LLM-generated corrections achieve 77% similarity to human developer fixes, making the auto-correction loop practical rather than theoretical.

**The implementation plan spans 8 weeks in phased delivery.** Layer 1 ships in weeks 1–2 and provides immediate value as a PR gate. Layer 2 follows in weeks 2–4 with Appium automation. Layer 3 completes in weeks 4–7 with the multimodal LLM analyzer and correction loop. Estimated ongoing cost is approximately `screens × scale_levels × $0.02` per analysis run, reducible through hash-based change detection. Key decisions still needed: confirming the mobile framework in use (React Native, Flutter, or native), Axe DevTools Mobile licensing, and whether a device cloud is required versus local emulators.
