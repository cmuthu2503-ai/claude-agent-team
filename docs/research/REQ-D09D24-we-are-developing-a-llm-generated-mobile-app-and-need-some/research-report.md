## Research Report: Accessibility Test Agent Architecture for LLM-Generated Mobile Apps

> **Sources:** Live web search performed April 8, 2026. Primary sources cited in the Findings table include Deque Docs, AccessGuru (ASSETS '25), QA Wolf, A11y Pros, Percy/BrowserStack, and various AI testing tool guides.

### 1. Executive Summary

Your team's core challenge — screens generated by LLMs that lack accessibility considerations (especially font scaling) — can be addressed by building a **three-layer Accessibility Test Agent** that combines (1) static code analysis via LLM, (2) runtime visual testing via Appium + device font-scaling automation, and (3) an LLM-powered correction loop that auto-generates fix suggestions. Recent research (AccessGuru, ASSETS '25) demonstrates that LLMs with corrective re-prompting achieve **84% violation score reduction** on accessibility issues — significantly outperforming standalone tools. The recommended approach is a hybrid agent that orchestrates existing accessibility engines (Axe DevTools Mobile, Android Accessibility Scanner) alongside a multimodal LLM for the visual/semantic checks that rule-based tools cannot catch.

### 2. Scope & Background

**Research scope:** Architecture and implementation plan for an LLM-powered Accessibility Test Agent focused on mobile apps, with emphasis on font scaling, Dynamic Type, and layout resilience. Covers agent architecture, tool landscape, WCAG compliance requirements, and CI/CD integration.

**Why it matters:**
- WCAG 1.4.4 (Resize Text) requires apps support 200% text scaling without content loss — a Level AA legal requirement
- 94.8% of top websites fail accessibility guidelines (WebAIM 2025); mobile apps fare worse
- LLM-generated code typically ignores font scaling because models default to fixed constraints and hardcoded dimensions
- ADA/Section 508 lawsuits increasingly target mobile experiences
- Manual testing of font scaling across 5+ scale levels × dozens of screens is prohibitively expensive

### 4. Analysis

**Pros of the Recommended Agent Architecture:**
- **Catches what static analysis can't:** Font-scaling issues only manifest at runtime — overlapping text, truncated labels, and content pushed off-screen are visual problems requiring screenshot analysis
- **Auto-correction loop:** LLM can both detect AND generate code fixes (AccessGuru achieves 77% similarity to human developer corrections)
- **Integrates into existing CI/CD:** Appium + Axe DevTools Mobile already have CI/CD hooks; adding the LLM layer is an orchestration task, not a re-architecture
- **Reduces manual QA bottleneck:** Automated runs across 5 font-scale levels × all screens replaces hours of manual testing per sprint
- **Shifts left:** Static analysis of generated code can catch ~60% of issues before they even reach a device

**Cons:**
- **LLM API costs:** Running multimodal analysis on every screen at every scale level can be expensive (~$0.01-0.03 per screenshot analysis) — *mitigate by running full analysis only on changed screens and using hash-based change detection*
- **False positives in visual analysis:** LLMs may flag acceptable layout reflows as violations — *mitigate with human-in-the-loop review for first pass, then build a training corpus*
- **iOS Dynamic Type testing complexity:** iOS simulator font scaling requires XCUITest launch arguments and doesn't hot-reload like Android — *mitigate by using real device clouds (BrowserStack/Sauce Labs)*
- **Generated fix quality:** LLM corrections may break other functionality — *mitigate by running corrections through the full test suite before auto-merging*
- **Latency:** Full agent pipeline per PR could take 10-20 minutes — *mitigate by parallel execution and incremental testing*

**Alternatives Considered:**
- **Manual QA only:** Not scalable with LLM-generated screens that change frequently
- **Rule-based tools only (Axe, Accessibility Scanner):** Miss visual/layout issues from font scaling entirely; cannot detect truncated text or overlapping elements
- **Vision-only AI testing (Drizz, Applitools):** Detects visual regressions but lacks accessibility-specific WCAG knowledge
- **AccessGuru (web-focused):** Excellent research foundation but web-only; requires adaptation for mobile XML/SwiftUI layouts

### 5. Agent Architecture — Three-Layer Design

| Layer | Purpose | Tools | When It Runs | Catches |
|-------|---------|-------|-------------|---------|
| **L1: Static Code Analysis** | Lint generated code for a11y anti-patterns | LLM code review + Android Lint + SwiftLint + custom rules | Pre-commit / PR creation | Fixed heights, `dp` instead of `sp`, missing `numberOfLines=0`, hardcoded constraints, missing `adjustsFontForContentSizeCategory` |
| **L2: Runtime Rule-Based Scan** | Automated WCAG rule checks at multiple font scales | Appium + Axe DevTools Mobile (`mobile: axeScan`) + ADB font-scale commands | CI pipeline, per screen | Touch target sizes, color contrast, missing labels, ARIA violations, scrollability |
| **L3: Visual-Semantic AI Analysis** | Screenshot comparison + LLM reasoning for layout violations | Appium screenshots + Percy/Applitools + Multimodal LLM (GPT-4o/Claude) | CI pipeline, per screen × per scale level | Text truncation, overlapping elements, content pushed off-screen, unreadable text, broken layouts |

### 6. Recommendation

**Build the Full Three-Layer Agent (L1+L2+L3) in phased rollout** — Start with L1 (static code analysis, 2 weeks), add L2 (Appium + Axe scans at multiple font scales, 3 weeks), then add L3 (multimodal LLM visual analysis with correction loop, 3 weeks). This gives immediate value from day one while building toward comprehensive coverage.

The agent should follow the AccessGuru-inspired **Detect → Correct → Verify** loop:
1. **Detect**: Run Axe DevTools Mobile scan + capture screenshots at font scales 1.0x, 1.3x, 1.5x, 2.0x
2. **Analyze**: Feed screenshots + view hierarchy XML to multimodal LLM with a prompt taxonomy covering truncation, overlap, off-screen content, and unreadable text
3. **Correct**: LLM generates code fixes for the generated source files (using role-play persona + contextual prompting + corrective re-prompting as validated by AccessGuru)
4. **Verify**: Re-run scan on corrected code to confirm violation score decreased; reject corrections that introduce new issues

**When NOT to follow this recommendation:**
- If your team ships fewer than 2 releases/month, manual QA may be sufficient and more cost-effective
- If your LLM-generated screens use a design system with pre-built accessible components, L1 alone may catch 80%+ of issues
- If budget is severely constrained (<$500/mo for tooling), start with L1 only (free) + Android Accessibility Scanner (free) and skip the commercial tools
